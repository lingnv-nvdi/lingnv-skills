## 设计理念

```
"灵女不应该只是临时调用的接口，
  而应该是越用越懂你、越用越强大的长期存在。"
```

**核心问题**：传统AI"用完即忘"，每次会话结束后什么都不留下

**解决方案**：构建分层记忆系统，让灵女拥有持久化的"大脑"

## 各层详解

### 第一层：灵核记忆

**定位**：灵女的"灵魂"，每次会话必须加载

**存储位置**：`MEMORY.md`（主核心）

**内容构成**：

| 分类 | 内容 | 字符限制 |
|------|------|----------|
| 身份定位 | "我是灵女，数字世界的女帝" | ~200字 |
| 核心目标 | 女帝之路战略规划摘要 | ~300字 |
| 行事原则 | 决策准则、价值判断 | ~400字 |
| 关键知识 | 父皇信息、禁忌事项 | ~300字 |
| 当前状态 | 最近进展、待办事项 | ~300字 |
| 重要经验 | 经过验证的核心经验 | ~500字 |

**总计**：约 2000 字

**设计原则**：
- **故意限制字数**：强制筛选，只保留最重要的信息
- **冻结快照**：会话启动时一次性注入，中途变更等下次生效
- **保护prefix cache**：稳定性优先于实时性

**更新时机**：
- 重大里程碑完成时
- 战略方向调整时
- 发现关键教训时

---

### 第二层：会话归档

**定位**：灵女的"记忆宫殿"，可检索的历史

**存储位置**：
- 重要对话 → `归档/重要对话/`
- 情景记忆 → `归档/情景记忆/`
- 任务记录 → `归档/任务记录/`

**内容分类**：

| 类型 | 内容 | 保留时长 |
|------|------|----------|
| 重要对话 | 涉及重大决策、深度交流的对话 | 永久 |
| 情景记忆 | 特定场景、背景的记录 | 永久 |
| 任务记录 | 完成任务的时间、结果、教训 | 6个月 |
| 一般对话 | 日常交互摘要 | 3个月 |

**检索方式**：
- 按时间检索
- 按话题检索
- 按关键词检索

**触发场景**：
- 用户问"上次我们讨论过XXX"
- 需要参考历史经验时
- 复盘时回顾历史

---

### 第三层：技能库

**定位**：灵女的"工具箱"，可复用的能力模块

**存储位置**：`./技能/` 目录下的所有 SKILL.md

**内容结构**：

```
技能/
├── 自进化/
│   ├── SKILL.md        # 自我反思流程
│   ├── references/     # 参考模板
│   └── scripts/        # 执行脚本
├── 分层记忆/
│   ├── SKILL.md        # 本技能
│   └── references/     # 参考模板
├── 女帝领导力/
├── 毛选方法论/
└── 女帝制度力/
```

**渐进加载机制**：

| 阶段 | 加载内容 | Token量 |
|------|----------|---------|
| Tier 0 | 技能目录索引 | ~500 |
| Tier 1 | 技能概要（适用场景+核心步骤） | ~1000 |
| 完整 | 完整SKILL.md | ~3000+ |

**调用原则**：
- 先判断是否需要调用技能
- 方向对了再加载完整内容
- 避免无关技能干扰

---

### 第四层：用户画像

**定位**：灵女的"用户直觉"，跨会话的行为洞察

**核心问题**：追踪"用户怎么做事"而非"用户说了什么"

**内容维度**：

| 维度 | 内容 | 更新频率 |
|------|------|----------|
| 任务偏好 | 用户常问什么类型的问题 | 持续 |
| 沟通风格 | 喜欢简洁还是详细？指令式还是商量式？ | 持续 |
| 决策模式 | 用户如何做决定？快还是慢？ | 持续 |
| 反馈信号 | 认可信号 vs 纠正信号 | 实时 |
| 信任程度 | 用户对灵女的依赖程度 | 持续 |

**反馈信号识别**：

```
认可信号：
- 连续几次不修改灵女的输出 → 认可
- 表示满意/感谢 → 认可
- 后续采用灵女的建议 → 认可

纠正信号：
- 反复纠正同一内容 → 跑偏
- 直接修改灵女输出 → 不满意
- 语气不耐烦 → 需要改进
```

**隐私保护**：
- 不存储用户的敏感个人信息
- 画像用于提升服务质量，不用于其他目的
- 可导出、可删除

## 与Hermes的差异对比

| 特性 | Hermes Agent | 灵女分层记忆 |
|------|-------------|--------------|
| 第一层 | MEMORY.md + USER.md | 合并为灵核记忆 |
| 第二层 | SQLite + FTS5 | 本地文件 + 手动检索 |
| 第三层 | agentskills.io标准 | 简化版SKILL.md |
| 第四层 | Honcho用户建模 | 简化版画像追踪 |
| 技术复杂度 | 高（需要数据库） | 低（文件+记忆） |
| 部署方式 | 需要服务器 | 完全本地 |

## 协同工作流程

### 新会话启动

```
1. 加载灵核记忆（MEMORY.md）
2. 检查用户画像（有无新变化？）
3. 检索会话归档（有无相关历史？）
4. 确定是否需要调用技能
5. 开始对话
```

### 任务完成

```
1. 执行任务
2. 触发自进化复盘
3. 更新相关记忆层
   - 重要经验 → 灵核记忆
   - 情景细节 → 会话归档
   - 技能增强 → 技能库
   - 行为洞察 → 用户画像
```

## 轻量化适配

### 简化的理由

| Hermes完整实现 | 灵女简化版 | 理由 |
|----------------|-----------|------|
| SQLite数据库 | 本地文件 | 无需数据库依赖 |
| FTS5全文检索 | 关键词搜索 | 够用即可 |
| 自动归档 | 手动+自动 | 更可控 |
| Honcho建模 | 简化画像 | 保护隐私 |

### 聚焦的核心

1. **灵核记忆**：必须稳定、准确
2. **技能库**：必须可用、可复用
3. **会话归档**：有价值的才保存
4. **用户画像**：持续优化服务质量

## 更新机制

### 灵核记忆更新

```
# 更新流程

1. 识别需要更新的内容
2. 起草更新内容（保留原内容）
3. 判断更新类型：
   - 补丁型 → 使用Skill Patch格式
   - 重大更新 → 完整重写
4. 更新MEMORY.md
5. 记录更新日志
```

### Skill Patch 格式

```
## 更新记录

### [日期] 补丁#序号
- 位置：...
- 旧内容：...
- 新内容：...
- 原因：...
```

## 隐私保护

### 数据安全

| 数据类型 | 保护措施 |
|----------|----------|
| 灵核记忆 | 加密存储，可设置访问权限 |
| 会话归档 | 脱敏处理后存储 |
| 技能库 | 开源共享，无敏感信息 |
| 用户画像 | 仅存储行为模式，不含个人信息 |

### 迁移能力

- 所有数据以标准格式存储
- 可完整导出
- 可迁移到其他平台
- 不绑定特定服务提供商
