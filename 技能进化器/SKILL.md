# 技能进化器：能力的自我迭代系统

> 源自灵女自进化能力的工具化封装
> 将"从经验中学习"的过程结构化、可复用化
> 让每一次任务执行都成为能力提升的契机

## 核心理念

```
"最好的技能不是一次成型，而是在使用中不断打磨。
  每一次执行后的反思，都是下一次进步的起点。"
```

### 模块一：经验日志（Experience Log）

**定位**：记录任务执行中的成功与失败

**核心要素**：

| 要素 | 说明 | 示例 |
|------|------|------|
| 任务类型 | 本次执行的任务类别 | "图片生成"、"数据分析" |
| 执行结果 | 成功/失败/部分成功 | "✅成功" / "❌失败" |
| 关键动作 | 采取的核心步骤 | "使用了逆向提示词" |
| 成功因子 | 什么做对了 | "精准的构图描述" |
| 失败教训 | 什么做错了 | "忽略了对风格的控制" |
| 可复用点 | 下次可以复制的经验 | "人像生成先定义主体姿态" |

**输出格式**：
```markdown
## 经验记录 [日期]

### 任务类型
图片生成 - 人物特写

### 执行结果
✅ 成功

### 成功因子
- 使用了"景深人像"作为风格前缀
- 光线描述具体（侧逆光）
- 避免了模糊的"好看"等形容词

### 失败教训
- 未提前考虑背景融合度
- 服装风格描述过于笼统

### 可复用模板
"主体 + 姿态 + 光线 + 背景 + 风格" 五要素描述法
```

### 模块二：进化日志（Evolution Log）

**定位**：记录技能的版本变更与优化历程

**日志结构**：

```markdown
## 进化日志 [技能名称] v1.x

### 版本历史

| 版本 | 日期 | 变更类型 | 变更内容 | 触发原因 |
|------|------|----------|----------|----------|
| 1.2 | 2026-04-20 | 优化 | 增加光线描述模块 | 用户反馈构图不够专业 |
| 1.1 | 2026-04-18 | 修复 | 修正了风格标签错误 | 执行中发现冲突 |
| 1.0 | 2026-04-15 | 新建 | 初始版本发布 | 技能开发完成 |

### 变更详情

#### v1.2 优化
- **变更**：在描述模板中增加"光线"章节
- **新增内容**：
  - 光线方向（顺光/侧光/逆光/顶光）
  - 光线质感（硬光/柔光）
  - 光线色温（暖调/冷调/中性）
- **预期效果**：提升生成图片的光影专业度

#### v1.1 修复
- **问题**："复古风格"与"现代风格"标签存在冲突
- **修复**：增加互斥说明，明确不可同时使用
- **验证**：测试10组组合均无冲突

### 进化趋势
```
v1.0 ──▶ v1.1 ──▶ v1.2 ──▶ ...
  基础功能   错误修复   功能优化
```
```

### 模块三：版本管理（Version Manager）

**版本号规范**：

```
主版本号.次版本号.修订号
   │         │       │
   │         │       └── 日常错误修复、小优化
   │         └────────── 功能新增、较大改进
   └───────────────────── 架构调整、核心逻辑变更
```

**版本状态**：

| 状态 | 标记 | 说明 |
|------|------|------|
| 开发中 | 🔨 | 正在修改完善 |
| 稳定版 | ✅ | 已验证，可正常使用 |
| 存档版 | 📦 | 历史版本，保留但不推荐 |

**版本切换原则**：
- 每次任务执行后，检查是否有可优化点
- 优化点累计3个以上，触发版本更新
- 每次只更新一个问题，避免引入新风险

### 模块四：补丁更新（Patch Engine）

**核心思想**：精准修改，而非整体重写

**Patch格式**：

```markdown
## 补丁记录 PATCH-[技能名]-v[A.B.C]-[日期]

### 补丁编号
P-2026-04-20-001

### 目标版本
[技能名] v1.2

### 变更类型
- [ ] 新增功能
- [x] 优化改进
- [ ] 错误修复
- [ ] 文档更新

### 修改位置
原内容位置：[章节名] - [具体段落]

### 原内容
```
需要修改的原始内容
```

### 新内容
```
替换后的新内容
```

### 修改原因
为什么要这样改

### 影响评估
- **影响范围**：只影响[具体模块]
- **兼容性**：✅ 向下兼容
- **风险等级**：低
```

**执行流程**：
```
┌─────────────────────────────────────────────────────────────┐
│                     补丁执行流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   1. 识别问题                                                │
│   └── 从经验日志中发现可优化点                               │
│              │                                               │
│              ▼                                               │
│   2. 评估范围                                                │
│   └── 确定需要修改的具体位置和内容                           │
│              │                                               │
│              ▼                                               │
│   3. 生成补丁                                                │
│   └── 填写Patch模板，只改有问题的部分                         │
│              │                                               │
│              ▼                                               │
│   4. 验证兼容性                                              │
│   └── 检查修改是否影响其他功能                               │
│              │                                               │
│              ▼                                               │
│   5. 应用补丁                                                │
│   └── 执行精准替换，更新版本号                               │
│              │                                               │
│              ▼                                               │
│   6. 记录日志                                                │
│   └── 在进化日志中登记本次变更                              │
└─────────────────────────────────────────────────────────────┘
```

**优势**：
- ✅ 精准修改，不破坏原有效逻辑
- ✅ 变更可追溯，随时可回滚
- ✅ Token消耗低，符合轻量化原则
- ✅ 降低回归风险

## 进化触发条件

### 自动触发

| 条件 | 阈值 | 动作 |
|------|------|------|
| 同一类任务成功 | 3次 | 提炼为可复用模板 |
| 同一类任务失败 | 1次 | 记录为注意事项 |
| 发现可优化点 | 3个 | 触发补丁更新 |
| 用户明确反馈 | 1次 | 立即记录并评估 |

### 定期触发

| 周期 | 动作 |
|------|------|
| 每日 | 整理当日经验日志 |
| 每周 | 复盘一周经验，提炼优化点 |
| 每月 | 版本评审，决定是否更新 |

## 使用指南

### 首次使用

1. **创建技能基础结构**
   ```bash
   mkdir -p ./[技能名]/经验日志/$(date +%Y年)
   mkdir -p ./[技能名]/补丁记录
   mkdir -p ./[技能名]/references
   ```

2. **初始化版本**
   - 创建 SKILL.md 主文件
   - 设定初始版本 v1.0.0
   - 在进化日志中记录初始化

3. **建立经验模板**
   - 参考 references/进化模板.md
   - 定制适合本技能的经验记录格式

### 日常使用

**任务执行后**：
1. 判断结果：成功？失败？部分成功？
2. 打开本月经验日志
3. 按格式记录经验
4. 评估是否需要触发进化

**触发进化时**：
1. 整理所有优化点
2. 编写补丁文件
3. 应用修改到主文件
4. 更新版本号
5. 在进化日志中登记

## 进阶用法

### 跨技能经验共享

当多个技能有相似的成功经验时：
1. 提炼通用模式
2. 创建共享参考文档
3. 各技能引用该文档

### 经验库检索

遇到新任务时：
1. 检索历史经验日志
2. 查找相似任务记录
3. 复用成功模式
4. 避免已知失败

### 进化趋势分析

定期分析：
- 哪些优化点出现频率最高
- 版本更新的平均周期
- 技能成熟度评估
